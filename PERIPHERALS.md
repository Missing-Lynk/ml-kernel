# Peripheral architecture: SoC + RF chip

Which hardware blocks on the Proxima-9311 SoC + AR8030 RF chip work with mainline/stock kernel drivers, and which needed an Artosyn-specific driver written for this project. "Stock" here means an upstream Linux driver already handles the IP block once it has a DT node; "custom" means a driver was written from scratch (usually recovered from the vendor 4.9 kernel) because no upstream driver exists for that SoC-specific block. The SoC/RF pair shows up across goggle, VRx, VTx, and air-unit products; hardware validation here happened on a goggle (the only variant this repo has hardware for), but the driver stack is the same across products - the panel/button/backlight rows below are goggle/VRx-specific (a display-and-keypad unit), while status LED, SD card, SDIO/RF, and the video codec engines apply to any Proxima-9311 product that has them. Source of truth for driver code: `modules/` (out-of-tree, `=m`), `overlay/` (built-in drivers we own, `=y`), and `patches/` (downstream diffs to mainline files, `=y`). On-device smoke tests: `test_tools/`.

**Current status of every row below: `STATUS.md`.** This doc is architecture only (what a peripheral needs and why it's stock vs custom) - it does not track progress.

## Proxima-9311 SoC peripherals

| Peripheral | Driver | Stock or custom |
|---|---|---|
| Status LED | mainline `dw_spi_mmio` (DesignWare SPI master) + mainline `spidev`; WS2812-style frames bit-banged from **userspace** | **Stock** - no kernel driver was written; the LED needed only a DT node (`spi@1102000` + spidev child). See `docs/status-led.md`. |
| Front-panel buttons | mainline `adc-keys` + mainline IIO core, fed by custom **`artosyn_adc`** (the SAR ADC IIO provider) | **Custom provider, stock consumer** - the ADC IP itself (`artosyn,adc` @ `0x0a108000`) has no upstream driver; `adc-keys` only needs a standard IIO channel to decode the resistor-ladder voltages into evdev key events. Register protocol: `docs/artosyn-adc.md`. |
| SoC temperature sensor | custom **`artosyn_protemp`** (IIO_TEMP provider) | **Custom** - the `artosyn,protemp` block (@ `0x0a108000`, shares the SAR ADC's MMIO window but is a separate engine: temperature code at `0xcc`/`0xd0`, linear code-to-Celsius) has no upstream driver. Register protocol: `docs/artosyn-protemp.md`. |
| Buzzer + LCD backlight | custom **`artosyn_pwm`** (the PWM chip provider) driving mainline `pwm-backlight` (LCD) and raw `/sys/class/pwm` (buzzer) | **Custom provider, stock consumer** - no upstream driver for the Artosyn PWM controller; once it registers a `pwmchip`, the mainline consumers work unmodified. |
| Display controller (VO + MIPI-DSI + panel) | fully custom **`artosyn_vo`** (DRM CRTC/display-controller), **`artosyn_dsi`** (glue feeding mainline `dw-mipi-dsi`), **`panel-qy45043a0`** (panel driver); DRM/KMS core itself is mainline, built as modules | **Custom** - the whole display pipeline is Artosyn-specific silicon; only the generic DRM/KMS core and the `dw-mipi-dsi` bridge are upstream. See `docs/display-backlight.md`. |
| SD card (microSD, `mmc1`) | mainline `dw_mci` core + custom **`dw_mci-artosyn`** glue (Artosyn SDMMC clock-tap/phase registers) | **Custom glue, stock core** - the DesignWare MMC/SD host IP is mainline, but this SoC's clock-select/phase registers (`0x0A108088`/`0x8C`) aren't the generic `snps,dw-mshc` binding. |
| RTC (external DS1307 on I2C0 @0x68) | mainline `i2c-designware-platform` (the `snps,designware-i2c` controller) + mainline `rtc-ds1307` | **Stock** - no code written; needed only the two DT nodes (copied from the vendor DTS) + a config fragment (`configs/i2c-rtc.config`). This is also exactly what the vendor does: its kernel drives the DS1307 with the same mainline driver and never binds any driver to the SoC-internal `rtc@0a10c000` block (no such symbols in its kallsyms), so the internal RTC is unused even on stock firmware. |
| CPU PMU (perf counters) | mainline `ARM_PMUV3` / `arm_pmu` | **Stock** - no code was written; needed only the `arm,cortex-a53-pmu` DT node (per-core overflow interrupts SPI 109/110, taken from the vendor DTS). Validated by `test_tools/pmu_test.c` (counting accuracy, overflow-IRQ routing per core). |
| Clock controller (CGU) | custom **`clk-ar9311-cgu`** (CCF provider, built in; replaced `clk-ar9311-pixel`) | **Custom** - the AR9311 CGU has no upstream driver and even the vendor kernel only models the CPU composite (everything else is driven from closed userspace via /dev/mem). The open driver registers the full 49-leaf tree read-only (register map recovered from the vendor `libmpp_service.so`) with exactly two settable clocks, both hardware-validated: the pixel PLL (panel-refresh steering) and `cgu_cpu_clk` (cpufreq/DVFS, all 8 vendor OPPs PMU-verified). Further write paths (per-clock gating of unused domains) are added one validated clock at a time. |
| Video codec (H.264/H.265, Chips&Media WAVE521C at `0x0a080000`) | mainline **`wave5`** V4L2 stateful M2M driver (=m) + downstream patches (`patches/0004`..`0011`: mmz pool routing, close-path leak fixes, sec-AXI off, sizeimage clamps, page-granular pool) | **Stock core, patched** - decode and encode both run through standard V4L2; only the VPU microcode (`chagall.bin`) stays closed. See `STATUS.md` "Open codec" + `docs/wave5-codec-capabilities.md`. The MPP `.ko` reimplementations are reference only (not shipped; pivot note in `modules/VERIFICATION.md`) - the vendor userspace they served is retired. |

## RF chip (AR8030)

| Peripheral | Driver | Stock or custom |
|---|---|---|
| SDIO device (`mmc0`, host link to the AR8030 baseband) | mainline `dw_mci` core + the same custom **`dw_mci-artosyn`** clock glue as the SD card, plus custom **`artosyn_gpio`** (drives the AR8030's active-low reset line, GPIO23) and custom **`artosyn_sdio`** (chip-specific firmware uploader + on-wire IP-over-SDIO protocol) | **Custom** - the SDIO host clock glue and GPIO reset are SoC-specific (see SD card row); `artosyn_sdio` has no upstream equivalent at all (it replicates the vendor's firmware-loader framing and `chagall` baseband wire format). |

## Reading this table alongside the code

- "Custom" modules live in `modules/*.c` (built out-of-tree, loaded from the rootfs) except the display stack, which is baked in via `overlay/drivers/gpu/drm/artosyn/` and `devices/betafpv-vr04-goggle/proxima-9311.dts` (see `README.md` "Built-in vs out-of-tree modules").
- "Stock" pieces are enabled purely through `configs/*.config` fragments (e.g. `spi.config`, `input.config`, `display.config`) - no source was written for them, only Kconfig symbols + DT nodes.
- The one recurring lesson across every custom driver: register maps recovered from disassembly must be confirmed bit-for-bit against a live vendor system (slot A) before trusting them - two of the hardest bugs found this way (`artosyn_gpio`'s bank-register off-by-one, and a clock-block register that should never have been touched) were single-register mistakes that looked plausible but silently broke a downstream peripheral.
